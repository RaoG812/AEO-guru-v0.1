Target Problem

Brands can't explain their AI answer footprint

Opportunity gap

Before this repo, we hand stitched crawl dumps, schema notes, and export scripts. The current codebase documents the exact operational gaps we had to close.

  • Crawls arrive from root URLs, sitemaps, and manual lists; app/api/ingest must juggle all three input types per run.

  • Analysts need deterministic text slices; lib/chunker.ts hashes each page + index so updates can be diffed.

  • Clusters require auditable context; the Qdrant payload stores clusterLabel, clusterSummary, and score_opportunity for every point.

20-page crawl window

crawlSite defaults to 20 URLs per run so live demos finish reliably.

1200 char chunks

chunkText slices content into 320-1200 character blocks for Gemini.

350ms crawl delay

The ingest worker intentionally pauses between requests to avoid hammering targets.

Operational guardrails

Limits + env vars keep runs predictable

Controls

Instead of imaginary projections, here are the concrete safeguards in this repo that keep usage sane when demoing ingestion → clustering → export loops.

Ingest guardrails

/api/ingest

Request schema enforces tight bounds.

  • Requires at least one of rootUrl, sitemapUrl, or urls[] before running.
  • Limit knob capped at 60 pages; crawl worker throttles at 350ms per request.

Vector hygiene

lib/qdrant.ts

Collections + summaries stay bounded.

  • ensureCollection sets Cosine distance + keyword indexes once per env.
  • Vector summary route fetches at most 2k points + trims preview arrays to eight floats.

Environment config

README.md

Spells out the required keys.

  • Gemini: GOOGLE_GENAI_API_KEY / GOOGLE_API_KEY.
  • Qdrant: QDRANT_URL + QDRANT_API_KEY.
  • Supabase: NEXT_PUBLIC_SUPABASE_URL + NEXT_PUBLIC_SUPABASE_ANON_KEY.

All of these constraints live in the repo today — no imaginary pricing tables required.
